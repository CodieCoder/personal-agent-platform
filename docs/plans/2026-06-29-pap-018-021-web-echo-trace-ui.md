# PAP-018 to PAP-021 Web Echo Trace UI

Date: 2026-06-29
Ticket range: PAP-018 to PAP-021
Status: Accepted for implementation

## Summary

Implement the first TanStack Start web slice:

- A web app shell in `apps/web`.
- A root route that shows server health/status and runs the echo capability.
- A TanStack server function that validates echo form input and calls the shared runtime.
- An execution detail route that reads persisted SQLite trace data and renders ordered trace steps.

Build directly on PAP-012 to PAP-017. Do not add new runtime capabilities, storage tables,
approval flows, memory, tool runtime, Docker, worker, chat UI, or external services in this slice.

## Decisions

- Use React, TanStack Start, TanStack Router, and Vite.
- Keep SQLite, migration, runtime, and capability wiring in server-only web modules.
- Run committed SQLite migrations once per web server process before runtime/repository use.
- Use `createRuntime({ traceRepository, capabilities: [echoCapability] })` from `@pap/runtime`.
- Use `capability.echo` as the only web-executable capability.
- Return safe discriminated unions from server functions instead of throwing raw database/runtime errors
  into the browser.
- Display only safe trace metadata: execution IDs, capability IDs, statuses, timestamps, step names,
  step summaries, and safe error code/message fields.
- Keep the UI as a restrained local operations console, not a landing page or chat interface.

## Web Behavior

- `/` renders a root layout, server status summary, echo message form, pending state, success state,
  safe failure state, execution detail link, and recent executions list.
- The echo form submits a `message` string to a server function.
- The server function validates the form payload, executes `capability.echo`, and returns the
  execution ID after success.
- Invalid or whitespace-only messages return a safe validation error.
- Completed messages render the normalized echo output and `echoedAt` timestamp.
- `/executions/$executionId` loads trace data by execution ID on the server.
- Unknown execution IDs render a safe not-found state.
- Trace steps render in repository sequence order.
- Completed traces remain visible after browser refresh because they are read from SQLite.
